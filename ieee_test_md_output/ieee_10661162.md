_**Abstract**_ **—In the data analysis process, visualized data can help users gain better insights. To make it easier and faster for users to obtain visual charts from data, natural language interfaces for data visualization have emerged. Users only need to provide the visualization model with the data to be visualized and a description of their visualization needs, and the model will return a visual chart(NL2VIS). In real-world scenarios, most data is stored in relational databases. To visualize this data, it is first necessary to generate a structured query statements based on the user’s visualization requirements(NL2SQL), and then proceed with the subsequent visualization operations. This study breaks down the task of automatic visualization of tabular data in relational databases into three main steps: generating SQL, determining the chart type, and mapping data to visual channels. We utilize the Chain-of-Thought(CoT) technique of generative large language models to address the task of automatic visualization of tabular data. Finally, we evaluated our approach on the nvBench dataset, and the results show that CoT-based automatic visualization of tabular data performs well.** 

_**Index Terms**_ **—NL2VIS, NL2SQL, Chain-of-Thought, Large Language Model**

## II. RELATED WORK 

In order to facilitate data analysis and visualization, a series of tools such as Power BI [1], Tableau [24], and Redash 

[23] have emerged successively. Users can utilize these tools for data visualization, but they often require a significant amount of manual operation. Subsequently, both academic and industry began researching natural language interfaces for data visualization, making user operations more convenient. These natural language interfaces enable users, even those lacking data analysis and visualization experience, to generate charts through language descriptions. The research on natural language interfaces for visualization has gone through three stages: rule-based methods, such as NL4DV [2], Orko [3], Eviza [4], FlowSense [5], DataTone [6]; deep neural networkbased methods, such as ADVisor [7], Seq2vis [8], ncNet [9], RGVisNet [10]; and generative large language model based methods, such as Chat2Vis [11], Prompt4Vis [12], Mirror [13], ChartGPT [14], and LIDA [15]. At present, the development of automated data visualization has progressed to a stage where large-scale models are used to solve problems. Based on large language models, two mainstream solutions have emerged: directly generating target code and visualization abstract expressions. For example, Chat2Vis [11] directly generates Python programs, while Prompt4Vis [12] generates visualization abstract expressions.To further improve the accuracy of data visualization, researchers tend to decompose visualization tasks into several sub-tasks, with each subtask responsible for solving a specific problem. The results of each subtask are then inputted into subsequent sub-tasks. For example, both ChartGPT [14] and Prompt4Vis [12] decompose the visualization task into sub-tasks and then solve each sub-task one by one, ultimately achieving good results in completing the visualization task. 

Large language models possess powerful In-Contextual Learning abilities [16]. In order to further enhance their reasoning capabilities, researchers have investigated prompt techniques such as zero-shot [17] and few-shot [16]. Subsequently, researchers discovered a Chain-of-Thought [18] prompt approach, which mimics the human thinking process by solving problems step by step. In academia, based on the chain-of-thought, other variants have been studied, such as Contrastive Chain-of-Thought [19], and Least-to-Most [20], etc.

## IV. SOLUTION OF COT-VIS 

To enhance the performance of large language models on downstream tasks, there are typically two mainstream approaches: first, fine-tuning the model on a dataset specific to the downstream task; second, constructing prompts to leverage the model’s In-Context Learning(ICL) ability to solve problems. Fine-tuning a model demands significant computational power and high-quality downstream task datasets, making it highly resource-intensive. In contrast, prompt engineering is simple and easy to implement, as it does not require updating the model’s parameters. A well-crafted prompt can guide the model to generate highly accurate answers. Prompt engineering is currently a subject of extensive research in the academic community and has already achieved remarkable success in tasks such as commonsense reasoning, mathematical problemsolving, and symbolic reasoning [18]. 

Next, we will elaborate on how to use prompt techniques in large language models to address the task of automatic visualization of tabular data in relational databases. First, we will summarize the current prompt strategies employed to enhance the reasoning capabilities of large language models. Then, we will select an appropriate solution specifically for the task of automatic visualization of tabular data. 

## _A. Prompt Strategies for Large Language Models_ 

Large language models can leverage their ICL abilities to respond to user queries based on the provided prompt text. Depending on the number of examples given by the user, prompt strategies can be categorized into two types: zero-shot and few-shot styles. Zero-shot prompts do not provide the model with question-answer examples, requiring the model to respond directly to the query. Few-shot prompts, on the other hand, provide the model with several question-answer examples, which can further enhance the accuracy of the model’s responses. 

The user provides the model with question-answer pairs in the form _< q, a >_ . Here, _q_ represents the user’s question along with any additional information needed to solve the problem, and _a_ represents the model’s response. Typically, prompts in the _< q, a >_ format are referred to as standardprompt. Researchers have discovered that by adding reasoning steps _r_ , prompting the model to derive the answer step by step, the accuracy of the model’s responses can be significantly improved. Consequently, researchers proposed the _< q, r, a >_ format prompt and named it Chain-of-Thought(CoT) prompt. Usually, users need to manually create several CoT examples to enable the model to mimic this format to generate answers. This type of prompt is known as few-shot CoT prompting. Additionally, there is a method that does not require manually crafted prompts: by providing the model with ” _**Let’s think step by step.**_ ”, the model is induced to produce CoT responses. This is referred to as zero-shot CoT prompting. Empirical evidence shows that manually few-shot CoT prompts outperform zeroshot CoT prompts in terms of performance.CoT prompt breaks down problems step by step, solving them gradually, which enhances problem-solving capabilities compared to standard prompts.

## V. EXPERIMENT 

_a) Dataset:_ We chose nvBench [8] as the evaluation dataset, widely used in the field of data visualization. nvBench is a large dataset designed for complex and cross-domain NL2VIS tasks, covering 105 domains, supporting seven common types of visualizations (Bar, Line, Scatter, Pie, Stacked Bar, Grouping Line, Grouping Scatter) and containing 25,750 (NL, VIS) pairs. Following the experimental validation method outlined in the Prompt4Vis paper [12], we randomly selected 141 databases from nvBench and divided them into training, validation, and test sets in a ratio of 7:2:1. Specifically, the training set contains 98 databases, the validation set contains 14 databases, and the test set contains 29 databases. 

_b) Model and Methods:_ In this experiment, we selected the GPT-3.5-turbo model interface provided by OpenAI and set temperature=0. The comparative methods include: zero-shot prompt, few-shot prompt, zero-shot-CoT prompt, and fewshot-CoT prompt. In the zero-shot prompt method, the model directly responds based on the provided database schema information and user natural language description. The few-shot prompt method requires manually writing several examples to provide to the model before letting it respond. The Zero-shotCoT prompt method adds ”Let’s think step by step.” to induce the model to generate chain-of-thought responses based on the zero-shot prompt. The few-shot-CoT prompt method involves manually writing seven chain-of-thought examples (one for each type of chart) to provide to the model for its response. 

_c) Metrics:_ This study comprehensively evaluates the data accuracy (data acc), axis accuracy(axis acc), chart accuracy(chart acc), and overall accuracy(overall acc) of the data visualization system. Data accuracy refers to the execution accuracy of the SQL predicted by the model, calculated as the proportion of correctly executed results to the total number of results. Axis accuracy is the accuracy of mapping data to visual channels, calculated similarly as a proportion. Chart accuracy compares the predicted chart type by the model with the gold chart, also calculated as a proportion.

## VI. CONCLUSION 

This study divides the task of automatic visualization of tabular data in relational databases into three main steps: generating SQL, determining the chart type, and mapping data to visual channels. Using the Chain-of-Thought technique of large language models, we perform step-by-step reasoning for these three steps. Experimental validation demonstrates that the Chain-of-Thought technique can be effectively applied to the task of automatic data visualization, significantly improving its accuracy. However, we should also note that there is still considerable room for improvement in the accuracy of automatic data visualization tasks, and we will continue to explore ways to enhance this accuracy. Additionally, for the task of automatic visualization of tabular data, the only evaluation dataset available is nvBench. We look forward to the academic community producing more high-quality evaluation datasets.
